# lock-master

[EN](README.md) | [DE](README_de.md) | [ES](README_es.md) | [JA](README_ja.md) | **RU** | [ZH](README_zh-Hans.md)

**Портативная система блокировки файлов на основе конфигурации для координации многоагентных проектов.**

lock-master предоставляет лёгкий протокол блокировки без зависимостей, основанный на обычных текстовых файлах. Файл `LOCK*.txt` в директории проекта сигнализирует о том, что проект или компонент используется в данный момент — ни один агент, автоматизация или цикл не должны изменять эту область, пока существует действительная, не истёкшая блокировка.

---

## Возможности

- **Блокировка по области (scope):** `LOCK.txt` блокирует весь проект; `LOCK.<scope>.txt` блокирует компонент. Несколько агентов могут параллельно работать с разными областями одного и того же проекта.
- **Автоматическое истечение срока:** каждая блокировка имеет настраиваемую длительность `expires_after` (по умолчанию 24 часа). Скрипт очистки удаляет забытые блокировки.
- **Сканирование только для чтения:** `lock_scan.py` перечисляет все активные блокировки во всех настроенных корневых директориях, не изменяя никаких файлов.
- **Кэш Markdown:** `lock_scan.py --write-cache` записывает `LOCK-CACHE.md` для мгновенного просмотра статуса без сканирования.
- **Пробный прогон (dry-run):** `prune_stale_locks.py --dry-run` предварительно показывает, что будет удалено.
- **Без зависимостей:** используется только стандартная библиотека Python (3.10+).
- **На основе конфигурации:** все корневые директории, ограничения глубины, пропускаемые папки и цели кэша задаются в `lock_roots.json` — без жёстко прописанных путей в коде.

## Быстрый старт

### 1. Скопировать скрипты

```
lock_utils.py
lock_scan.py
prune_stale_locks.py
LOCK_TEMPLATE.txt
```

Поместите их в директорию по вашему выбору (например, `scripts/`).

### 2. Создать `lock_roots.json`

Скопируйте `lock_roots.example.json`, переименуйте его в `lock_roots.json` и замените пути-заглушки на реальные корневые директории проектов. Файл исключён из системы контроля версий через `.gitignore` (он содержит локальные абсолютные пути).

```json
{
  "default_max_depth": 4,
  "shallow_depth": 2,
  "skip_dirs": [".git", ".venv", "node_modules", "__pycache__", "build", "dist"],
  "roots": [
    { "path": "/путь/к/проекту-a" },
    { "path": "/путь/к/проекту-b" },
    { "path": "/путь/к/большому-дереву", "shallow": true }
  ],
  "caches": [
    {
      "name": "общесистемный",
      "path": "/путь/к/scripts/LOCK-CACHE.md"
    }
  ]
}
```

### 3. Создать блокировку

Скопируйте `LOCK_TEMPLATE.txt` в директорию проекта, заполните поля и переименуйте файл в `LOCK.txt` (или `LOCK.<scope>.txt` для блокировки уровня компонента):

```
owner: мой-агент
created: 2026-06-14T10:00
expires_after: 24h
mode: hard
purpose: Рефакторинг модуля аутентификации
```

### 4. Вывести список активных блокировок

```bash
python lock_scan.py
python lock_scan.py --json
```

### 5. Удалить истёкшие блокировки

```bash
# Предварительный просмотр (безопасно):
python prune_stale_locks.py --dry-run

# Фактическое удаление:
python prune_stale_locks.py
```

### 6. Обновить кэш

```bash
python lock_scan.py --write-cache
```

Записывает `LOCK-CACHE.md` в соответствии с настройками ключа `"caches"` в `lock_roots.json`.

## Формат файла блокировки

Обычный текст, по одной записи `ключ: значение` на строку. Строки, начинающиеся с `#`, являются комментариями.

| Поле                | Обязательное | Пример               | Значение |
|---------------------|--------------|----------------------|----------|
| `owner`             | да           | `мой-агент`          | Кто владеет блокировкой. |
| `created`           | да           | `2026-06-14T10:00`   | ISO-временная метка; основа для расчёта истечения срока. |
| `expires_after`     | необязательно | `24h`, `90m`, `2d`  | Строка длительности. По умолчанию: `24h`. |
| `release_condition` | необязательно | `PR смержен`        | Свободный текст: когда блокировку можно снять. |
| `mode`              | необязательно | `hard` \| `soft`    | `hard` = без изменений (по умолчанию); `soft` = чтение/подсказки допустимы. |
| `purpose`           | необязательно | `Добавление фичи X` | Описание выполняемой работы в свободной форме. |
| `scope`             | необязательно | `frontend`           | Информационное; **имя файла** является авторитетным источником. |

Если `created` отсутствует или не поддаётся разбору, в качестве запасного варианта используется mtime файла.

## Соглашение по областям (scope)

| Имя файла            | Определённая область | Что заблокировано |
|----------------------|----------------------|-------------------|
| `LOCK.txt`           | `project`            | Вся директория проекта |
| `LOCK.api.txt`       | `api`                | Только компонент `api` |
| `LOCK.frontend.txt`  | `frontend`           | Только компонент `frontend` |
| `LOCK.my_scope.txt`  | `my_scope`           | Любая произвольно названная подобласть |

Регулярное выражение для определения: `^LOCK(\.[^.]+)?\.txt$` (без учёта регистра).

## Жизненный цикл

```
УВАЖАТЬ  -->  ЗАНЯТЬ  -->  ОСВОБОДИТЬ
```

1. **УВАЖАТЬ:** перед началом работы над проектом или компонентом проверить наличие активного `LOCK*.txt`, охватывающего эту область. Если таковой существует и не истёк — выбрать другую задачу или подождать.
2. **ЗАНЯТЬ:** создать файл блокировки из шаблона (`owner`, `created`, `expires_after`, `purpose`).
3. **ОСВОБОДИТЬ:** **удалить собственный файл блокировки** по завершении работы. Активное освобождение является обязательным; таймаут `expires_after` — лишь страховочная сеть для забытых блокировок. Если работа занимает больше времени, чем ожидалось, обновите `created`, чтобы предотвратить преждевременное истечение срока.

## Справочник по конфигурации (`lock_roots.json`)

| Ключ                | Тип      | По умолчанию | Описание |
|---------------------|----------|--------------|----------|
| `default_max_depth` | int      | `4`          | Максимальная глубина рекурсии по директориям от каждого корня. |
| `shallow_depth`     | int      | `2`          | Глубина для корневых директорий, отмеченных `"shallow": true`. |
| `skip_dirs`         | string[] | `[]`         | Имена директорий для полного пропуска (включая поддерево). |
| `roots`             | object[] | `[]`         | Список `{ "path": "...", "shallow": true/false }`. |
| `caches`            | object[] | `[]`         | Цели кэша: `{ "name", "path", "filter_prefix?" }`. |

**Поля записи кэша:**

| Ключ            | Обязательное | Описание |
|-----------------|--------------|----------|
| `name`          | да           | Отображаемое имя, используемое как заголовок кэша. |
| `path`          | да           | Абсолютный путь, по которому записывается `LOCK-CACHE.md`. |
| `filter_prefix` | необязательно | Включать только блокировки, путь которых начинается с данного префикса. |

Если `"caches"` опущен, `--write-cache` записывает единственный `LOCK-CACHE.md` рядом с `lock_scan.py`.

## Python API

```python
from pathlib import Path
import lock_utils

project = Path("/путь/к/моему-проекту")

# Проверить перед началом работы
active = lock_utils.active_locks(project)
if active:
    print(f"Заблокировано: {active}")
else:
    print("Свободно для работы.")

# Разобрать конкретный файл блокировки
data = lock_utils.parse_lock_file(project / "LOCK.txt")
print(data["owner"], data["created"])

# Проверить истечение срока
from datetime import datetime
expired = lock_utils.is_expired(project / "LOCK.txt", now=datetime.now())
```
